# FRH Production Recovery Procedures

This document covers recovery from common production failures including P2021 errors, failed migrations, and database issues.

---

## P2021 "table does not exist" errors

A `P2021` error means Prisma tried to query a table that does not exist in the connected database. This is the most common production failure mode.

### Diagnosis steps

1. **Check if migrations were deployed:**
   ```bash
   # Requires DIRECT_URL set to production Supabase Session-mode string (port 5432)
   npx prisma migrate status
   ```
   If there are pending migrations, they were never applied to this database.

2. **Check if the correct Supabase project is targeted:**
   ```bash
   npm run verify:env
   ```
   This runs `scripts/verify-env.mjs` which extracts the project-ref from your URLs and confirms both `DATABASE_URL` and `DIRECT_URL` point to the same Supabase project. Compare the project-ref against your Supabase dashboard.

3. **Check if the database was wiped or recreated without re-running migrations:**
   ```bash
   node scripts/verify-db.mjs
   ```
   If all tables return errors, the database is empty or the connection string is completely wrong. If only some tables are missing, migrations are partially applied.

### Resolution

Once you have identified the cause:

- **Pending migrations:** Run `npx prisma migrate deploy` with your production `DIRECT_URL`.
- **Wrong project:** Update `DATABASE_URL` and `DIRECT_URL` in Vercel to point to the correct Supabase project. Redeploy.
- **Wiped database:** Run `npx prisma migrate deploy` to recreate all tables. If seed data is needed, see the Re-seeding section below.

## Recovering from failed migrations

If `prisma migrate deploy` fails with a drift error, the database schema differs from what the migration history records. This is typically caused by a previous `prisma db push` against the wrong database.

See the "Recovering from drift" section in `docs/PRISMA_WORKFLOW.md` for the full procedure. Summary:

```bash
# 1. Inspect what Prisma thinks differs
npx prisma migrate diff \
  --from-migrations prisma/migrations \
  --to-schema-datamodel prisma/schema.prisma \
  --script

# 2. If the diff is trivial, mark the drift as resolved by baselining:
npx prisma migrate resolve --applied <migration-name>

# 3. If the diff is complex:
#    a. Take a Supabase point-in-time backup
#    b. Test resolution on a staging copy first
#    c. Apply to production carefully
```

**Never run `prisma migrate reset --force` on production.** It drops all tables and data.

## Verify production DB has correct tables and data

Run the database verification script:

```bash
node scripts/verify-db.mjs
```

Expected output after a fresh seed:

| Table | Expected Count |
|---|---|
| Season | 1 |
| Player | 20 |
| Team | 10 |
| Division | 2 |
| God | 83 |
| HomepageContent | 0 (created on first admin edit) |

If counts are zero across all tables but no errors are thrown, the database has the correct schema but no data. See the Re-seeding section.

If specific tables throw errors, those migrations were never applied.

## Re-seeding the database

**WARNING: The seed script performs upserts but is designed for development databases. Running it against production with live data can overwrite real records. Never seed production if it contains live match data, standings, or player stats that were not generated by the seed.**

If you are certain the production database needs seed data (e.g., after a complete database recreation):

```bash
# 1. Verify migrations are applied first
npx prisma migrate status

# 2. Run the seed
node prisma/seed.mjs
```

The seed script:
- Inserts a Season, Divisions, Teams, Players, and Gods
- Is idempotent (uses upsert) for reference data
- Does NOT create matches, drafts, or stats

### When it is safe to seed production

- After a complete database recreation (new Supabase project, all data lost)
- After confirming no live operational data exists
- With explicit admin sign-off

### When it is NOT safe to seed production

- If any matches have been played and recorded
- If standings or stats contain real data
- If you are unsure whether live data exists

## Vercel environment variable verification checklist

Before each production deploy or after any credential rotation, confirm in Vercel > Project > Settings > Environment Variables:

- [ ] `DATABASE_URL` - Supabase Transaction URL, port 6543
- [ ] `DIRECT_URL` - Supabase Session URL, port 5432
- [ ] Both URLs have same project-ref and password
- [ ] `ADMIN_SESSION_SECRET` - at least 16 characters
- [ ] `ADMIN_PASSWORD` - set for admin dashboard access
- [ ] `ADMIN_AUTH_REQUIRED` - set to `true` for production
- [ ] `NEXTAUTH_URL` - set to production domain
- [ ] `GEMINI_API_KEY` - set if screenshot OCR is needed
- [ ] Discord env vars - set if Discord OAuth is required (see README.md)

### Quick validation after updating env vars

```bash
# Run verify:env against the new values locally
npm run verify:env

# Or trigger a Vercel redeploy and check the function logs for startup errors
```

## When to contact admin for manual intervention

Escalate to admin if:

- `prisma migrate status` shows a failed migration that cannot be resolved with `prisma migrate resolve`
- The Supabase project was deleted or is inaccessible
- The database password was rotated but you do not have the new one
- Data corruption is suspected (row counts are wrong, foreign keys are broken)
- A migration needs to be rolled back (Prisma does not support automatic rollback -- manual SQL is required)
- You need to restore from a Supabase point-in-time backup

For all manual interventions, gather this information first:
1. Output of `npx prisma migrate status`
2. Output of `node scripts/verify-db.mjs`
3. Output of `npm run verify:env`
4. The exact error message from the application logs
