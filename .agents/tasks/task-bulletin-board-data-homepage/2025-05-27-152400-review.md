# Bulletin Board system and data-driven homepage rebuild

The change replaces the old editorial homepage system (admin-editable fake content with a HomepageContent DB model) with two things: a data-driven league dashboard that renders real database content (standings, matches, drafts) and a new Bulletin Board feature providing admin-managed community posts with a full lifecycle (draft/published/archived). The homepage now shows a "Latest from the Bulletin Board" section linking the two pieces together.

**Watch for:** Slug collision on post creation (confirmed); PATCH endpoint regenerates slug on every title edit, breaking existing links (confirmed); no test coverage for any new bulletin board code (confirmed); admin auth on the bulletin board pages relies on a hand-rolled `fakeReq` object that is fragile (confirmed).

## High-level view

The homepage cleanup deletes ~1800 lines of editorial/editor scaffolding, replaced by 136 lines that derive all sections from real DB data. Power rankings are now computed from actual standings rather than a hardcoded roster of joke team names.

The Bulletin Board is a self-contained mini-CMS: a Prisma model with relations to the existing entity graph (Player, Team, Match, Division, Season), REST API routes with Discord-role auth gating, a listing page with admin inline controls, and a single-post detail page with inline editing.

Slug generation uses a random 4-character suffix for uniqueness, but has no retry mechanism if the slug already exists in the database. With very short suffixes and common titles, this will eventually collide and throw a unique-constraint error that surfaces as a generic 500 to the admin. Worse, the PATCH handler regenerates the slug on every title edit, breaking existing URLs.

The server-side admin detection in the page components constructs a synthetic request object to reuse `getDiscordSessionUser`. This is duplicated in two files and coupled to the internal cookie-parsing implementation in discordAuth.js — it would break silently if that library ever accesses additional request properties.

1. **Slug collision on create** — `generateSlug` appends a 4-char random suffix but never checks for existing slugs or retries. A collision returns a Prisma P2002 error surfaced as a 500. Add a retry loop or query-before-insert.
2. **PATCH regenerates slug on title edit** — Every title update generates a new random slug, breaking all existing links to the post (homepage preview, Discord embeds, shared URLs). 5. **Silent error swallowing on bulletin post fetch** — The homepage try/catch for `recentBulletinPosts` has an empty catch body (no `console.error`), unlike every other query on the page. Real DB errors in production will be invisible.

## Slug generation and collision risk

Both the list route (`POST /api/bulletin-board`) and the update route (`PATCH /api/bulletin-board/[id]`) use `generateSlug`:

```javascript
function generateSlug(title) {
  const base = title.toLowerCase().replace(/[^a-z0-9]+/g, '-').replace(/-+/g, '-').replace(/^-|-$/g, '');
  const suffix = Math.random().toString(36).slice(2, 6);
  return `${base}-${suffix}`;
}
```

The `slug` column has a `@unique` constraint in Prisma. If two posts titled "League Update" generate the same 4-char suffix (probability increases with post volume), the create call throws a P2002 error. The catch block in the POST handler returns a generic "Failed to create post" 500, giving the admin no indication that retrying would work.

More critically, the PATCH handler calls `generateSlug` whenever `body.title` is provided. This means editing a post's title also changes its slug, breaking any external links (homepage bulletin preview, Discord embeds, shared URLs) that used the original slug. The slug should be immutable after creation unless explicitly requested by the admin.

## Server-side admin detection pattern

The two page server components duplicate a `checkAdminFromCookies` function that builds a synthetic request object:

```javascript
const fakeReq = {
  headers: {
    get: (name) => {
      if (name === 'cookie') return `${DISCORD_SESSION_COOKIE}=${sessionCookie.value}`;
      return null;
    },
  },
};
```

This is duplicated verbatim in `bulletin-board/page.js` and `[slug]/page.js`. A shared helper like `getSessionFromCookieStore(cookieStore)` in `lib/discordAuth.js` that accepts the Next.js cookie store directly would eliminate both the duplication and the fragility of depending on the internal structure of `getDiscordSessionUser`.

## Auth: silent filter stripping

The GET handler for the list route silently ignores status filters from non-admins:

```javascript
if (statusFilter && admin) {
  where.status = statusFilter;
} else {
  where.status = 'published';
}
```

This is fail-closed (non-admins always get published posts regardless of what they request), but it means a non-admin client making a `?status=draft` request receives published posts with a 200. An explicit 403 would be more honest about what happened.

## Test coverage gap

The bulletin board feature adds zero tests. Given that the API routes handle auth checks, slug generation, status transitions, and data validation, at minimum the following are untested:

- Auth enforcement (can a non-admin POST/PATCH/DELETE?)
- Slug uniqueness behavior on collision
- Status transition logic (draft->published sets publishedAt, re-publish doesn't overwrite)
- Input validation (missing title, invalid type)
- Admin GET vs public GET filtering behavior

## Homepage silent error catch

The server component fetches bulletin posts with a try/catch that silently swallows errors:

```javascript
try {
  recentBulletinPosts = await prisma.bulletinPost.findMany({ ... });
} catch (err) { /* BulletinPost model not yet available */ }
```

The comment "not yet available" is a leftover from development iteration. Now that the migration is included in the same PR, the silent catch masks real database errors in production. Adding a `console.error` (matching the pattern used by every other try/catch on that page) would prevent silent failures.
